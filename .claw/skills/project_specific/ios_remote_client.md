# iOS Remote Client для Claw-Code - Технічне Завдання (Skill)

## 1. Загальний опис
Мета цього завдання — створити повноцінний управлінський фронтенд для `claw-code` на базі iOS-додатка (Remote Client). Цей додаток буде виступати клієнтом, а існуючий `claw-code` — бекендом.

Додаток повинен надавати:
1. Вікно чату для спілкування з агентом (надсилання промптів, отримання відповідей).
2. Інтерфейс перегляду виконання плану (Roadmap/Task Graph).
3. Сортування та фільтрацію по помилках (failed tasks, exceptions).
4. Перегляд та фільтрацію використаних інструментів (executed tools) під час сесії.
5. Загальний моніторинг робочого процесу (Status & Logs).

## 2. Основні екрани (UI/UX)
- **Chat View (Чат):** Класичний інтерфейс чату, де користувач може надсилати текстові повідомлення агенту і отримувати markdown-відповіді.
- **Task/Plan Dashboard (Вікно виконання плану):** Ієрархічне відображення `task.md` у вигляді дерева (Graph). Повинні бути чітко позначені виконані `[x]`, активні `[/]` та провалені `[-]` завдання. Можливість розгортати/згортати гілки плану.
- **Terminal/Logs View (Фільтрація логів):** Окремий екран для перегляду технічних логів з можливістю сортувати події за типами (помилки, виклики інструментів, системні повідомлення).

## 3. Мережевий рівень (Networking)
Додаток повинен підключатися до локального (або віддаленого) бекенду `claw-code`.
- **WebSocket:** Використовувати `URLSessionWebSocketTask` для забезпечення двостороннього з'єднання у реальному часі (Real-time).
- **JSON RPC/NDJSON:** Підтримка парсингу потокових даних, які повертає `claw-analog` або `claw-rag-service`.
- **Моделі даних (Data Models):**
  - `Message`: текстове повідомлення (користувач/агент).
  - `TaskNode`: елемент графа завдань зі статусами (`pending`, `in_progress`, `completed`, `failed`).
  - `ToolCall`: інформація про використаний інструмент та його результат (з підтримкою маркування помилок).

## 4. Вимоги до реалізації
- **Фреймворк:** SwiftUI.
- **Архітектура:** MVVM (Model-View-ViewModel).
- **Ізоляція стану:** Чітке розділення UI від логіки обробки WebSocket-повідомлень. Зміни в плані повинні автоматично відображатися на графіку (Reactive UI).

## 5. Перші кроки для розробки (Task Roadmap)
1. Створити базові моделі для парсингу WebSocket-повідомлень від `claw-code`.
2. Реалізувати WebSocket Manager для стабільного підключення та реконнекту.
3. Побудувати SwiftUI-компоненти для чату.
4. Розробити UI для Task Graph з відображенням ієрархії та статусів (включаючи гілки з провалами `[-]`).
5. Зв'язати фільтри логів (Errors, Tools) зі стрімом даних, що надходить по сокету.
